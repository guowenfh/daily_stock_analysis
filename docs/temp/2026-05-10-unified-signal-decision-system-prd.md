# 统一信号决策系统 - 产品需求文档 (PRD)

> **日期**: 2026-05-10
> **状态**: 草案
> **范围**: 将 Bilibili 信号提取、个人自选股分析、回测验证、决策支持统一为一个完整的投资决策产品。

---

## 1. 产品愿景

### 1.1 目标

构建一个统一的投资决策支持系统，实现：

1. **全量采集**：处理每一个关注 UP主 的每一条内容（视频、动态文本、图文、文章、转发）
2. **信号提取**：通过 LLM 提取所有资产类型的交易信号（A股个股、港股、美股、ETF、指数、板块、期货）
3. **共识聚合**：多 UP主 加权共识评分，含冲突检测
4. **准确性验证**：以 1/3/7 天窗口验证信号准确率
5. **融合决策**：将 Bilibili 情报与自选股分析、技术指标、新闻、基本面结合
6. **实时 + 日报**：高价值信号实时推送 + 每日综合决策报告
7. **辅助决策**：结构化的买入/卖出/持有建议

### 1.2 核心原则

**信号驱动分析**：Bilibili 内容不是一个独立功能，而是驱动整个分析管线的主要情报输入。当 3 个以上 UP主 看好同一只股票时，该股票应自动进入深度分析管线进行技术面/基本面评估。

### 1.3 用户故事

> 作为一名个人投资者，我关注了 60 多个 B站财经 UP主。他们每天发布数十条视频、动态和文章，讨论具体的股票、ETF、板块和期货。我希望系统能自动处理所有这些内容，提取每一个被提及的资产及其情绪和关键价位，展示哪些资产有多 UP主 的强共识，验证哪些 UP主 历史上预测准确，对高共识标的进行深度分析，并推送包含具体操作建议的日报——这样我就不需要手动观看 60 多个视频来做决策。

### 3.2 数据流（端到端）

```
1. [每小时] ContentFetcher 抓取所有 UP主 动态 → Content (processed=false)
2. [新内容到达] ExtractorRegistry 分发到对应类型的提取器
3. [每条内容] LLM 提取 StockMention[] + ContentAnalysis
4. [每条内容] 存储 TradeSignal + StockMention，标记 processed=true
5. [提取完成后] ConsensusEngine 聚合每只股票的共识
6. [提取完成后] SignalEventEngine 重建事件
7. [每日 15:30] UnifiedDecisionEngine 汇编：
   a. Bilibili 共识信号（按 UP主 准确率加权）
   b. 高共识股票的技术分析
   c. 新闻/基本面上下文
   d. 历史信号的回测验证
   e. 风险评估
   → 生成每资产的 UnifiedDecision
8. [每日 16:00] DailyReportGenerator 生成综合报告
9. [高分事件触发] RealTimeAlert 立即推送
10. [用户查询] DecisionDashboard 展示所有活跃决策
```

## 4. 功能需求

### 4.1 F1: 统一决策引擎

**优先级**: P0

核心新增组件。将所有情报源融合为每资产的决策。

#### 4.1.1 UnifiedDecision 数据模型

```python
class UnifiedDecision:
    asset_key: str           # 如 "600519", "ETF:510300", "FUT:rb2410"
    asset_name: str          # 资产名称
    asset_type: AssetType    # STOCK, ETF, INDEX, SECTOR, FUTURES, HK_STOCK, US_STOCK
    market: Market           # A_SHARE, HK, US
    decision_date: date      # 决策日期

    # 评分 (0-100)
    consensus_score: float       # Bilibili 多 UP主 共识评分
    technical_score: float       # 技术分析评分
    fundamental_score: float     # 基本面健康评分
    news_sentiment_score: float  # 新闻/搜索情绪评分
    backtest_trust_score: float  # 该资产/UP主 的历史信号准确率

    # 最终决策
    composite_score: float       # 加权综合评分
    direction: Direction         # STRONG_BUY, BUY, HOLD, SELL, STRONG_SELL
    confidence: float            # 0.0-1.0 置信度
    action_plan: str             # 结构化操作建议

    # 证据链
    uploader_signals: list       # 哪些 UP主 说了什么
    key_levels: KeyLevels        # 支撑位/阻力位
    risk_factors: list           # 风险因素
    backtest_summary: dict       # 历史准确率统计
```

#### 4.1.2 评分权重（可配置）

| 来源 | 默认权重 | 条件 |
|------|---------|------|
| Bilibili 共识 | 35% | 2+ UP主 同方向时生效 |
| 技术分析 | 30% | 始终生效 |
| 新闻/基本面 | 20% | 有数据时生效 |
| 回测信任度 | 15% | 有 8+ 条已验证信号时生效 |

当共识不足 2 个 UP主 提及时，权重重新分配：技术 (45%) + 基本面 (30%) + 新闻 (25%)。

#### 4.1.3 决策规则

- **强烈买入 (STRONG_BUY)**：composite_score >= 80 且 3+ UP主 看多 且 技术面确认上升趋势
- **买入 (BUY)**：composite_score >= 65 且 (共识看多 或 技术面看多)
- **持有 (HOLD)**：composite_score 40-65 或 信号矛盾
- **卖出 (SELL)**：composite_score <= 35 且 (共识看空 或 技术面看空)
- **强烈卖出 (STRONG_SELL)**：composite_score <= 20 且 3+ UP主 看空

### 4.2 F2: 全量内容覆盖

**优先级**: P0

每一个 UP主 的每一条内容都必须处理，不遗漏。

#### 4.2.1 内容类型

| 类型 | 当前处理方式 | 目标 |
|------|-------------|------|
| 视频 | 字幕提取 (API + Whisper) → LLM | 已确认可用，增加质量评分 |
| 动态文本 | 直接 LLM 提取 | 已确认可用 |
| 动态图文 | 多模态 LLM（K线图、截图） | **验证多模态调用是否工作，增加降级方案** |
| 文章 | 直接 LLM 提取 | 已确认可用 |
| 转发 | 提取原文 + 评论上下文 | 已确认可用 |

#### 4.2.2 处理保障

- **不跳过任何内容**：所有 `processed=false` 的 Content 记录都必须尝试处理
- **错误韧性**：提取失败记录错误但不阻塞管线
- **重试策略**：瞬态 LLM 失败重试 3 次，指数退避
- **质量评分**：每次提取根据字幕可用性、内容长度、LLM 置信度打分
- **死信队列**：失败 3+ 次的内容标记为需人工审查

#### 4.2.3 内容去重

当多个 UP主 讨论同一新闻/事件时：
- 按 (asset_key, 日期, 相似度哈希) 分组
- 不抑制——而是按 UP主 独立性加权（不同来源 = 更强信号）
- UI 显示"N 位 UP主 讨论了此事"

### 4.3 F3: 全资产类型支持

**优先级**: P0

从仅支持个股扩展到所有可交易资产类型。

#### 4.3.1 资产类型映射

| 资产类型 | 代码模式 | 行情数据源 | 备注 |
|---------|---------|-----------|------|
| A股个股 | 600xxx, 000xxx, 300xxx, 688xxx | Akshare, Efinance, Tushare | 已有 |
| 港股 | HK+5位 (如 HK00700) | Yfinance, Longbridge | 已有 |
| 美股 | 1-5字母代码 (如 AAPL) | Yfinance, Longbridge | 已有 |
| ETF | 510xxx, 159xxx, 513xxx | Akshare | **新增：ETF 代码识别** |
| 指数 | 000001 (上证), 399001 (深证), 000300 (沪深300) | Akshare | **新增：指数代码映射** |
| 板块 | 板块名 (如 "半导体", "新能源") | Akshare 板块数据 | **新增：名称到代码解析** |
| 期货 | rb2410, IF2410, au2412 | Akshare 期货数据 | **新增：期货代码识别** |

#### 4.3.2 LLM Prompt 增强

当前 `config/prompts/bilibili_signal.yaml` 中的 prompt 已要求输出 `asset_type`。需要增强：

- 在 prompt 中为每种资产类型添加明确示例
- 在提取器后处理中增加板块/ETF/期货代码解析逻辑
- 新增 `AssetCodeResolver` 服务，将 LLM 提取的名称映射到标准代码

#### 4.3.3 共识引擎扩展

当前 `ConsensusEngine` 按 `stock_code` 分组。扩展：
- 按 `asset_key` 分组（标准化：`{类型}:{代码}` 或 `{类型}:{名称}`）
- 支持跨资产类型共识（如 "半导体" 板块 + "中芯国际" 个股 + "半导体ETF" ETF）

### 4.4 F4: 实时信号推送

**优先级**: P0

高价值信号立即推送，而非仅在日报中体现。

#### 4.4.1 推送触发条件

| 触发条件 | 条件 | 紧急度 |
|---------|------|--------|
| **强共识** | 3+ UP主 24小时内看多同一资产 | 高 - 立即推送 |
| **高分事件** | SignalEvent 评分 >= 80 | 高 - 立即推送 |
| **分歧警报** | 2+ UP主 对同一资产观点矛盾 | 中 - 1小时内推送 |
| **新 UP主 信号** | 高权重 UP主 (>0.8) 的首个信号 | 中 - 1小时内推送 |
| **回测失败** | 之前推荐的信号出现亏损 | 低 - 纳入日报 |

#### 4.4.2 推送格式

```
[信号提醒] 600519 贵州茅台
方向: 买入（强共识）
评分: 85/100
UP主: 3/5 看多 (财经老王: 买入@1800, 股市小李: 买入@1780, 技术派老张: 买入@1820)
支撑位: 1750 | 阻力位: 1850
风险: 近期涨幅较大，注意回调
来源: [链接1] [链接2] [链接3]
```

#### 4.4.3 去重与限流

- 同一资产：每 4 小时最多 1 条推送
- 同一 UP主：每小时最多 3 条推送
- 全局：每小时最多 10 条推送（可配置）
- 摘要模式：1 小时内超过 5 条推送时，切换为批量摘要

### 4.5 F5: 每日决策报告

**优先级**: P0

综合所有情报源的每日报告。

#### 4.5.1 报告结构

```markdown
# 每日决策报告 - 2026-05-10

## 摘要
- 5 个新买入信号，2 个卖出信号，12 个持有
- 最佳机会：600519 (85/100, 3 位 UP主 看多)
- 最大风险：000858 (退出风险, 2 位 UP主 看空)

## 第一部分：高共识标的
| 资产 | 评分 | 方向 | UP主 | 关键价位 | 操作建议 |
|------|------|------|------|---------|---------|
| 600519 | 85 | 买入 | 3/5 | 支撑: 1750 | 现价可入，止损1720 |

## 第二部分：信号事件
- [机会] 半导体板块 - 4 位 UP主 看多，评分 78
- [退出风险] 000858 五粮液 - 2 位 UP主 看空，评分 35
- [分歧] 601318 中国平安 - 2 看多 vs 2 看空

## 第三部分：自选股分析
（来自个人自选股的完整分析）
- 600519: 技术上升趋势 + 共识看多 = 强烈买入
- AAPL: 技术中性 + 无共识 = 持有

## 第四部分：回测验证
- 过去 7 天：12 条信号已验证，8 条正确（67% 准确率）
- 最佳 UP主：财经老王（80% 准确率，10 条信号）
- 最差 UP主：股市新手（40% 准确率，5 条信号）

## 第五部分：市场环境
- A股：上证 +0.5%，成交量上升 15%
- 关键事件：央行降准，美联储会议纪要

## 第六部分：操作清单
1. 买入 600519 @ 1780-1800，止损 1720，止盈 1850
2. 关注 半导体板块 突破
3. 若跌破 150 则减持 000858
```

#### 4.5.2 报告生成管线

```
每日 15:30（A股收盘后）：
1. 运行 UnifiedDecisionEngine 处理所有活跃资产
2. 获取市场综述（A股/港股/美股）
3. 运行 1/3/7 天前信号的回测验证
4. 汇编报告各部分
5. 生成 Markdown + 飞书富文本卡片 + 可选图片
6. 推送到所有已配置的通知渠道
7. 保存到 data/reports/{date}.md
8. 更新 Web 端决策仪表盘
```

### 4.6 F6: 回测与准确率闭环

**优先级**: P1

闭环：信号 → 验证 → 调整权重 → 更好的信号。

#### 4.6.1 信号验证流程

```
信号创建（TradeSignal 含 signal_date, stock_code, sentiment）
    │
    │ +1天 / +3天 / +7天
    v
SignalVerificationScheduler：
    1. 获取 signal_date + N 天的实际价格
    2. 计算 change_pct
    3. 判断方向是否正确
    4. 更新 TradeSignal (price_after_1d/3d/7d, is_verified, verification_result)
    5. 插入 SignalVerification 记录
    6. 更新 UploaderSignalConfig.historical_accuracy（当已验证信号 >= 8 条时）
```

#### 4.6.2 每位 UP主 的准确率指标

| 指标 | 计算方式 | 用途 |
|------|---------|------|
| 1日准确率 | correct_1d / verified_1d | 短期可靠性 |
| 3日准确率 | correct_3d / verified_3d | 中期可靠性 |
| 7日准确率 | correct_7d / verified_7d | 长期可靠性 |
| 平均置信度 | mean(confidence) | 自评校准 |
| 平均收益率 | mean(change_pct) | 盈利能力 |

#### 4.6.3 权重反馈循环

```
UploaderSignalConfig.final_weight = base_weight * time_decay * accuracy_multiplier

accuracy_multiplier:
  - 8+ 条已验证信号：将准确率从 [0.3, 0.9] 映射到 [0.7, 1.3]
  - < 8 条信号：1.0（中性）
  - 上下限：[0.5, 1.5]，防止极端权重
```

### 4.7 F7: 决策仪表盘（Web）

**优先级**: P1

一个展示"今天该怎么做？"的网页。

#### 4.7.1 仪表盘布局

### 4.8 F8: 问股集成

**优先级**: P1

当用户执行 `/ask 600519` 时，响应应包含 Bilibili 信号上下文，与策略分析并列。

#### 4.8.1 AskCommand 增强

当前流程：股票代码 → 加载策略 → Agent 使用技术工具分析
增强流程：股票代码 → 加载策略 → **注入 Bilibili 共识上下文** → Agent 分析

Agent 的上下文应包含：
- 最近有多少 UP主 提及了该股票
- 汇总情绪（看多/看空/中性）
- UP主 提及的关键价位
- UP主 共识评分
- 该股票的历史信号准确率

#### 4.8.2 实现方式

在 `bot/commands/ask.py` 中，构建 Agent 上下文前：
1. 查询该股票的 `ConsensusResult`（最近 7 天）
2. 查询最近的 `StockMention` 记录，含 UP主 名称和情绪
3. 查询是否存在活跃的 `SignalEvent`
4. 作为"Bilibili 情报"部分注入 Agent 的 prompt

使 `/ask` 命令成为所有情报源（技术分析 + Bilibili 信号 + 新闻 + 基本面）的汇聚点。

### 4.9 F9: UP主 准确率排行榜

**优先级**: P1

#### 4.9.1 展示指标

- 总体准确率（正确数 / 已验证总数）
- 按时间窗口的准确率（1天、3天、7天）
- 每条信号的平均收益率
- 信号数量

#### 4.9.2 权重调整可见性

## 8. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM 幻觉（编造股票名称） | 产生虚假信号 | 将提取的代码与行情数据交叉验证；拒绝未知代码 |
| 字幕质量下降 | 遗漏信号 | 多级降级（API → Whisper → 标题）；质量评分 |
| UP主 合谋（讨论同一消息） | 共识膨胀 | 按 UP主 独立性加权；检测协同行为 |
| Bilibili 限流 | 遗漏内容 | 多种抓取方式（bili CLI, API）；错峰抓取 |
| LLM 成本激增 | 预算超支 | 每次会话 Token 预算；批量处理；模型分级 |
| SQLite 并发问题 | 高负载下数据损坏 | 如并发写入成为瓶颈，迁移到 PostgreSQL |
