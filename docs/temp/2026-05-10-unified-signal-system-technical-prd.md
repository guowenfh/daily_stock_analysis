# 统一信号系统 PRD

> 日期：2026-05-10  
> 状态：草案  
> 文档定位：产品 PRD / 开发对齐文档  
> 目标读者：产品、设计、后端、前端、测试、新技术团队  
> 范围：把财经内容创作者的观点，稳定转换为可追溯、可解释、可筛选的单标的投资信号。

---

## 1. 背景

当前项目已经具备股票分析、报告生成和部分内容处理基础，但新的信号系统需要重新作为一条独立主链路来建设。

这条链路的核心不是一开始就做复杂交易决策，而是先把“UP 主说了什么、提到了什么标的、观点强弱如何、证据在哪里、哪些内容失败了为什么失败”做清楚。

V1 先聚焦 Bilibili。未来会扩展到小红书、微博等平台，但所有平台内容都必须先转换成统一内容形态，再进入同一套信号提取和看板流程。

---

## 2. 产品目标

### 2.1 一句话目标

把分散在 Bilibili UP 主内容里的股票、ETF、指数、板块观点，转换成可追溯的结构化信号，并在前端提供采集质量、处理队列、信号看板和单标的详情。

### 2.2 V1 需要回答的问题

- 今天系统采集了多少内容？
- 哪些 UP 主覆盖到了，哪些没有覆盖到？
- 哪些内容成功提取了信号？
- 哪些内容失败了，失败原因是什么？
- 哪些标的被 UP 主提到？
- 每个标的的观点来自哪些 UP 主、哪些内容、哪些原文证据？
- 同一个标的是否形成机会、风险、分歧或观察信号？
- 用户配置的高权重 UP 主是否影响信号排序？

### 2.3 V1 重点指标

| 指标 | 说明 |
| --- | --- |
| 内容数 | 指定时间范围内采集到的内容总量 |
| 内容提取成功率 | 成功完成信号提取的内容占比 |
| UP 主覆盖率 | 已启用 UP 主中，本轮成功采集到内容或确认无新增内容的占比 |
| 失败可解释率 | 失败内容中，有明确失败阶段和可读失败原因的占比 |

## 3. 用户与使用场景

### 3.1 主要用户

系统的主要用户是个人研究者，关注财经 UP 主观点，希望用系统辅助整理信息，而不是替代自己的判断。

### 3.2 核心使用场景

**每日复盘**

用户每天打开系统，先看今天采集是否正常，再看今日机会、风险和分歧。用户不需要逐条刷动态，而是先看系统整理出的标的和证据。

**单标的追踪**

用户对某个标的感兴趣时，进入标的详情页，查看最近有哪些 UP 主提到它、观点方向是什么、原文证据是什么、是否存在相反观点。

**UP 主权重管理**

用户对部分 UP 主有长期观察经验，可以手动配置更高权重。系统在排序和事件评分中体现该权重，但必须保留证据链，不能让权重覆盖事实。

**失败排查**

当内容没有被提取时，用户需要知道是采集失败、字幕缺失、图片无法识别、LLM 输出异常，还是标的无法解析，并能决定是否重试或忽略。

## 4. V1 范围

### 4.1 必须支持

- Bilibili UP 主内容采集。
- UP 主列表、启用状态和人工权重配置。
- 三类统一内容形态：
  - 纯文本。
  - 图文。
  - 视频与视频字幕。
- 专栏按图文内容处理。
- 内容处理状态可见。
- 信号提取结果可追溯到 UP 主、内容和原文证据。
- 单标的信号聚合。
- 机会、风险、分歧、观察四类事件。
- 采集质量看板。
- 内容处理队列。
- 信号看板。
- 标的详情页。
- 创作者权重页面。

### 4.2 暂不支持

- 小红书、微博完整接入；后续优先级为小红书高于微博。
- 自动准确率计算和自动权重调整。
- 自选股分析、技术面、新闻、基本面融合。
- 板块、指数、ETF、期货之间的跨资产联动。
- 真实持仓和仓位联动。
- 自动交易。
- 复杂回测、RAG、桌面端或部署系统改造。
- 额外免责声明或合规流程。

## 5. 内容与信号链路

### 5.1 总链路

```text
配置 UP 主
  -> 采集 Bilibili 内容
  -> 转换为统一内容形态
  -> 补全字幕、图片文字、引用上下文
  -> 提取标的与观点
  -> 归并同一标的
  -> 应用人工 UP 主权重
  -> 生成机会 / 风险 / 分歧 / 观察事件
  -> 在前端看板和每日摘要中展示
```

### 5.2 内容形态标准化

系统不应该让后续模块直接依赖 Bilibili 的私有字段。所有内容进入信号提取前，都要转换为以下三类形态。

| 内容形态 | 来源举例 | 处理方式 |
| --- | --- | --- |
| 纯文本 | 动态正文、转发正文、短文本观点 | 直接提取文本观点和标的 |
| 图文 | 图片动态、专栏、长文、图文笔记 | 合并正文、图片文字、图片描述后提取 |
| 视频与视频字幕 | 视频标题、简介、字幕、转录文本 | 基于标题、简介、字幕提取，不应只看标题 |

未来小红书、微博也必须先转换到这三类形态，再进入同一条信号链路。

### 5.3 内容处理状态

用户不需要看到工程内部细节，但必须能理解内容卡在哪一步。

| 状态 | 用户理解 |
| --- | --- |
| 已采集 | 系统拿到了原始内容 |
| 待补全 | 需要补字幕、图片文字或引用上下文 |
| 待提取 | 内容已准备好，等待信号提取 |
| 已提取 | 已生成结构化信号 |
| 低置信 | 有结果，但质量不足，需要复核 |
| 失败 | 某一步失败，需要展示原因 |

### 5.4 失败原因展示

失败必须可解释。前端至少需要展示：

- 失败阶段：采集、标准化、补全、提取、标的解析。
- 失败原因：例如登录失效、内容不存在、字幕缺失、图片无法解析、LLM 输出格式错误、标的无法识别。
- 建议动作：重试、等待下次采集、人工忽略、人工复核。

## 6. 信号定义

### 6.1 单条信号

单条信号表示“某个 UP 主在某条内容中提到了某个标的，并表达了某种观点”。

一条信号至少需要包含：

- 标的名称。
- 标的类型：股票、ETF、指数、板块。
- 市场：A 股、港股、美股或未知。
- 观点方向：看多、看空、中性。
- 置信度。
- 是否为主标的。
- 原文证据。
- 操作建议原文摘要；没有明确建议时不得编造。
- 关键价位；没有明确价位时为空。
- 来源 UP 主。
- 来源内容。
- 原文链接。

### 6.2 信号质量

系统需要对信号质量做标记，而不是只给出结果。

常见质量标记：

- 字幕缺失。
- 只基于标题提取。
- 图片文字不完整。
- 标的代码未识别。
- 标的名称存在歧义。
- LLM 置信度低。
- 原文没有明确操作建议。

### 6.3 单标的事件

事件是对同一标的多条信号的聚合结果。

V1 只做单标的事件，不做跨资产联动。

| 事件类型 | 触发含义 |
| --- | --- |
| 机会 | 多个信号偏正向，或高权重 UP 主明确看多 |
| 风险 | 多个信号偏负向，或高权重 UP 主明确提示风险 |
| 分歧 | 同一标的同时存在明显多空观点 |
| 观察 | 有提及但强度不足，或者信息不完整 |

事件必须能展开看到每条底层信号，不能只展示一个总结分数。

## 7. 前端页面设计

### 7.1 信息架构

V1 建议提供五个主页面：

```text
信号总览
采集质量
内容队列
标的详情
UP 主管理
```

### 7.2 信号总览

这是用户每天最先看的页面。

**页面目标**

让用户快速知道今天哪些标的值得看、哪些标的有风险、哪些标的观点冲突。

**核心区域**

- 顶部指标：
  - 今日新增内容数。
  - 提取成功率。
  - UP 主覆盖率。
  - 失败可解释率。
- 事件分组：
  - 今日机会。
  - 今日风险。
  - 今日分歧。
  - 今日观察。
- 标的事件列表：
  - 标的名称。
  - 事件类型。
  - 方向。
  - 分数或强度。
  - 涉及 UP 主数。
  - 信号数。
  - 最高权重 UP 主。
  - 最新更新时间。

**交互**

- 点击事件进入标的详情。
- 支持按事件类型、市场、标的类型、UP 主筛选。
- 支持只看高权重 UP 主参与的事件。
- 支持隐藏低置信事件。

### 7.3 采集质量

**页面目标**

让用户判断系统今天是否正常工作。

**核心区域**

- 内容采集趋势。
- 各 UP 主采集状态。
- 内容形态分布：纯文本、图文、视频字幕。
- 处理漏斗：已采集、待补全、待提取、已提取、低置信、失败。
- 失败原因排行。

**关键指标**

- 内容数。
- 内容提取成功率。
- UP 主覆盖率。
- 失败可解释率。

**交互**

- 点击失败原因进入内容队列。
- 点击某个 UP 主查看其采集记录。
- 支持按时间范围筛选。

### 7.4 内容队列

**页面目标**

让用户处理未完成、低质量或失败内容。

**列表字段**

- 发布时间。
- UP 主。
- 内容标题或摘要。
- 内容形态。
- 当前状态。
- 失败原因或质量标记。
- 是否已产生信号。
- 操作入口。

**操作**

- 查看原文。
- 查看提取证据。
- 重试。
- 标记忽略。
- 标记需要人工复核。

**筛选**

- 待补全。
- 待提取。
- 失败。
- 低置信。
- 已提取。
- 内容形态。
- UP 主。

### 7.5 标的详情

**页面目标**

让用户围绕一个标的做判断。

**核心区域**

- 标的基础信息。
- 当前事件类型：机会、风险、分歧、观察。
- 多空观点比例。
- 涉及 UP 主列表。
- 高权重 UP 主观点。
- 关键价位。
- 信号时间线。
- 原文证据列表。

**信号卡片**

每张信号卡片展示：

- UP 主名称和权重。
- 内容标题。
- 发布时间。
- 观点方向。
- 置信度。
- 原文证据。
- 操作建议摘要。
- 关键价位。
- 原文链接。
- 质量标记。

**交互**

- 按 UP 主筛选。
- 按观点方向筛选。
- 只看高置信信号。
- 展开原文证据。
- 从信号跳转到原始内容。

### 7.6 UP 主管理

**页面目标**

让用户维护采集对象和人工权重。

**字段**

- UP 主名称。
- 平台。
- 平台账号 ID。
- 是否启用。
- 分类。
- 人工权重。
- 擅长市场。
- 擅长资产类型。
- 最近采集时间。
- 最近提取成功率。
- 备注。

**操作**

- 新增 UP 主。
- 启用或停用。
- 修改人工权重。
- 修改分类和备注。
- 查看该 UP 主最近内容和信号。

**权重规则**

- V1 只支持用户手动配置权重。
- 系统可以展示历史表现参考，但不自动改权重。
- 权重影响事件排序和强度，但不隐藏相反证据。

## 8. 每日使用流程

### 8.1 正常流程

1. 用户打开信号总览。
2. 查看顶部采集质量指标。
3. 浏览今日机会、风险、分歧和观察。
4. 点击某个标的进入详情。
5. 查看涉及 UP 主、原文证据、观点方向和关键价位。
6. 如发现内容缺失或失败，进入内容队列排查。
7. 如发现某个 UP 主长期更可靠，在 UP 主管理中调整人工权重。

### 8.2 失败排查流程

1. 用户在采集质量页看到失败数升高。
2. 点击失败原因进入内容队列。
3. 查看失败阶段和失败原因。
4. 对可恢复失败执行重试。
5. 对不可恢复失败标记忽略或等待下次采集。

### 8.3 单标的研究流程

1. 用户从信号总览或搜索进入标的详情。
2. 查看多空观点比例。
3. 查看高权重 UP 主是否参与。
4. 展开信号卡片查看原文证据。
5. 对比相反观点和质量标记。
6. 用户自行决定是否纳入自己的研究判断。

## 9. 平台扩展原则

V1 只实现 Bilibili，但产品设计必须为后续平台保留空间。

后续平台优先级：

1. 小红书。
2. 微博。

扩展原则：

- 新平台必须接入同一套内容形态：纯文本、图文、视频与视频字幕。
- 新平台不能绕过统一信号提取链路。
- 前端筛选需要支持平台维度。
- 同一标的可以来自多个平台，但 V1 先不要求跨平台共识。
- 平台差异只影响采集和内容补全，不影响最终信号展示结构。

## 10. 验收标准

### 10.1 采集质量

- 可以维护 Bilibili UP 主列表。
- 可以采集启用 UP 主的最新内容。
- 可以展示内容数、提取成功率、UP 主覆盖率、失败可解释率。
- 单个 UP 主失败不影响其他 UP 主。
- 失败内容有明确阶段和可读原因。

### 10.2 内容处理

- 纯文本、图文、视频字幕能进入统一处理流程。
- 专栏按图文处理。
- 字幕缺失、图片无法识别等情况会进入质量标记或失败原因。
- 用户可以在内容队列中查看、筛选、重试和忽略。

### 10.3 信号提取

- 信号能展示标的、观点、置信度、证据、来源 UP 主和原文链接。
- 没有明确操作建议时，不生成虚假操作建议。
- 未识别标的不能进入高置信事件。
- 低质量内容不会被包装成高置信信号。

### 10.4 事件聚合

- 同一标的多条信号可以聚合成事件。
- 事件能区分机会、风险、分歧、观察。
- 人工高权重 UP 主会影响排序。
- 事件能展开到底层信号和原文证据。
- V1 只做单标的事件，不做跨资产联动。

### 10.5 前端体验

- 用户可以从信号总览进入标的详情。
- 用户可以从采集质量进入内容队列。
- 用户可以从标的详情跳转原始内容。
- 用户可以配置 UP 主人工权重。
- 用户可以理解失败原因和下一步动作。

## 11. 分阶段交付

### M1：采集质量可见

目标：用户能知道系统采集是否正常。

交付：

- UP 主管理。
- Bilibili 内容采集。
- 采集质量页。
- 内容数、UP 主覆盖率、失败可解释率。

### M2：内容队列可用

目标：用户能看到每条内容的处理状态。

交付：

- 纯文本、图文、视频字幕三类内容形态。
- 内容队列。
- 失败原因和质量标记。
- 重试、忽略、人工复核入口。

### M3：信号看板可用

目标：用户能看到结构化标的信号。

交付：

- 信号总览。
- 单条信号卡片。
- 标的、观点、证据、来源 UP 主、原文链接。
- 低置信标记。

### M4：标的详情和事件可用

目标：用户能围绕单个标的做研究。

交付：

- 标的详情页。
- 机会、风险、分歧、观察事件。
- 多空观点比例。
- 高权重 UP 主观点。
- 信号时间线。

### M5：每日摘要

目标：用户每天可以用摘要快速复盘。

交付：

- 今日机会摘要。
- 今日风险摘要。
- 今日分歧摘要。
- 采集质量摘要。
- 可追溯到标的详情和原始内容。

## 12. 后续阶段

### V2：Bilibili 链路稳定化

- 提升字幕、图片文字、专栏处理质量。
- 优化低置信和人工复核流程。
- 优化每日摘要。

### V3：多平台扩展

- 优先接入小红书。
- 再接入微博。
- 增加平台筛选和平台来源展示。

### V4：决策融合

- 将内容信号接入自选股分析。
- 融合技术面、新闻、基本面。
- 支持问股上下文和提醒。

V4 之前不做完整交易决策系统。
